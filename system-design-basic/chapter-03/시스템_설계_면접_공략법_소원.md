# 1. 설계 면접 개요

- **단순 설계 능력만 보는 것이 아님** 설계의 순수성에 집착해 **오버엔지니어링**을 하는지, 비용 대비 효율을 고려하는지 체크함
- 완고하고 편협한 사고를 가졌는지, 비즈니스 요구사항에 따른 **Trade-off**를 유연하게 받아들이는지 확인
- 이상적인 설계 능력뿐만 아니라, 회사의 이익을 이해하고 면접관과 협력하여 최적의 해를 찾아가는 '동료로서의 자질'을 증명해야 함

---

# 2. 4단계 면접 Step

### **Step 1. 문제 이해 및 설계 범위 확정 (3~10분)**

- **요구사항 명확화**: 바로 설계를 시작하지 말고, 요구사항과 가정을 반드시 확인.
- **요구사항 기록**: 화이트보드나 노트에 핵심 요구사항을 작성하여 시각화.
- **비기능적 요구사항 확인**: 사용자 수(DAU), 응답 지연 시간, 데이터 보관 기간, 가용성 등 시스템의 규모와 제약 사항을 수치로 정확히 파악.

### **Step 2. 개략적인 설계안 제시 및 동의 구하기 (10~15분)**

- **핵심 컴포넌트를 포함한 다이어그램을 그리고 1차 동의를 얻어야 함. 디테일하게 말고 대략적으로 설계를 함**
    - 구성 요소: 클라이언트, API, WAS, DB, Cache, CDN, MQ 등..
- **추정치 계산**: 2장에서 다룬 개략적 추정이 필요한 단계인지 면접관에게 확인
- **면접관의 의견을 묻고 고려해야할 부분들 추가**

### **Step 3. 상세 설계 (10~25분)**

- **우선순위 선정**: 전체 시스템에서 가장 중요하거나 병목이 예상되는 부분에 집중
- 뉴스피드 시스템이라면 '피드 발행'과 '피드 생성' 등 핵심 비즈니스 로직과 그 관련된 것들에 한 단계 더 깊게 파고들 것

### **Step 4. 마무리 (3~5분)**

- 자신의 설계에 남아있는 약점을 스스로 언급하고 개선 방향을 제시하는 비판적 사고가 필요.
- 설계의 핵심 포인트와 결정 이유를 요약해서 다시 한번 정리
- 오류 처리, 로그 수집, 모니터링, 배포 전략 등 실무적인 디테일을 언급하며 운영적 측면도 같이 이야기하면 좋음
- 시스템 규모가 10배, 100배 커질 때 발생할 문제와 대처 방안을 가볍게 언급하면 좋음

# 3. 체크리스트
- 해야 할 것
    1. 질문을 통해 요구사항을 명확히 할 것 
    2. 정답이 없음을 인지하고 요구사항에 집중할 것 
    3. 여러 대안을 제시하고 각각의 장단점을 설명할 것 
    4. 사고 과정을 말로 표현할 것
    
- 하지 말아야 할 것
    1. 가정 없이 성급하게 설계를 시작하는 것 
    2. 소통 없이 혼자서만 설계하는 것 
    3. 막혔을 때 힌트 요청을 주저하는 것 
    4. 구체적인 기술(특정 DB 제품명 등)에 너무 매몰되는 것
